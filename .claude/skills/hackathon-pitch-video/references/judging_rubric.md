# Judging rubric — and how the video addresses each axis

**Always use the hackathon's real rubric if one exists.** Ask for it. If none is published,
this 4×25% set is the common default (used by ACM-style TechSprint / Asteria events and many
others). Each axis below lists what the judge is scoring and the **beat that earns it** so the
video covers every criterion on purpose, not by accident.

## Standard 4×25% rubric

| Criterion | Weight | What the judge scores | Video beat that earns it |
|---|---:|---|---|
| Innovation & Creativity | 25% | Is the core idea original / non-trivial, or a wrapper around an existing tool? | Beat 3 (the clever bit) + the hook |
| Technical Execution | 25% | Does it actually work, end to end? Is it reproducible / well-built? | Beat 2 (live demo) + Beat 3 |
| Design & User Experience | 25% | Is it usable, legible, well-presented? Docs/onboarding? | Beat 2 (demo legibility) + video production quality itself |
| Theme Relevance & Impact | 25% | Does it hit the challenge theme? Real-world, reusable impact? | Beat 1 (problem) + Beat 4 (impact/fit) |

> Note: the **video's own production quality is itself a Design/UX signal.** A clean, legible,
> well-paced video raises the Design score even before judges open the repo. Sloppy audio or
> unreadable screen capture costs you here.

## Turning the rubric into a coverage check

Before recording, confirm every criterion has a moment in the script that earns it:

- [ ] **Innovation** — there is a clear sentence naming the one thing only this project does.
- [ ] **Technical Execution** — the video *shows the thing running*, not slides describing it.
- [ ] **Design/UX** — the demo is legible (big fonts, zoom in) AND the video is clean.
- [ ] **Theme/Impact** — the challenge theme is named in the judges' own words, with who it helps.

If a criterion has no beat, the project will score low on it regardless of how good the work is —
judges score what they *see*, not what's in the repo.

## Weighting your time to the weighting

If all four axes are 25%, don't spend 80% of the video on the demo and 5 seconds on impact.
Rough time allocation for a 3:00 video on the default rubric:

- Hook + Problem (Theme/Impact setup): ~0:30
- Live demo (Technical + Design): ~1:15
- The clever bit (Innovation + Technical): ~0:30
- Impact + fit (Theme/Impact payoff): ~0:25
- Limitations + next step (credibility): ~0:15
- Close: ~0:05

Adjust to match the **actual** weights if the hackathon weights criteria unequally — put video
minutes where the points are.

## If the rubric differs

Common variants to watch for and the beat to add:
- **Business viability / market** → add a one-line "who pays / how it scales" to Beat 4.
- **Use of a specific sponsor tech / API** → show it on screen in the demo and name it.
- **Presentation/pitch quality scored separately** → tighten delivery, add captions, rehearse.
- **Social impact** → make Beat 1 (problem) and Beat 4 (impact) the emotional center.
